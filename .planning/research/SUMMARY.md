# Project Research Summary

**Project:** Terraform Provider for Pure Storage FlashBlade ‚Äî milestone v2.1.1 (Network Interface / VIP)
**Domain:** Terraform provider (Go) ‚Äî REST API wrapping for enterprise storage appliance
**Researched:** 2026-03-30
**Confidence:** HIGH

## Executive Summary

This milestone adds `flashblade_network_interface` (VIP) resource and data source management to an existing, production-grade Terraform provider for Pure Storage FlashBlade REST API v2.22. The provider already implements 29+ resources following a well-established internal pattern (framework v1.19, separate POST/PATCH/GET structs, `?names=` query-param identity, three-tier test coverage). The VIP resource fits this pattern with one critical difference: VIP `name` is auto-generated by the array (marked `ro` in the API spec), not user-specified. Every design decision flows from this constraint.

The recommended approach is pure mechanical extension of existing patterns: new `models_network.go` + `network_interfaces.go` in the client layer, new resource and data source files in the provider layer, and additive (read-only) enrichment of the existing server resource/data source. No new dependencies are required. The eight-step build order (models ‚Üí client ‚Üí mock ‚Üí resource ‚Üí data source ‚Üí server enrichment √ó 2 ‚Üí registration) has hard dependency ordering at steps 1-3, with parallel execution possible from step 4 onward.

The primary risks are specific to VIP semantics: treating `name` as user-specified, reusing GET structs for PATCH bodies, and accidentally granting the server resource write authority over VIP attachments. A secondary risk is the schema version bump required on `flashblade_server` ‚Äî adding a computed attribute without a `StateUpgrader` entry breaks existing users on provider upgrade. All risks are fully understood and mitigated by established patterns already present in the codebase.

## Key Findings

### Recommended Stack

The stack is unchanged from the base provider and requires no new `go.mod` entries. All required primitives (HTTP client, NamedReference structs, list handling, drift logging) already exist. The milestone is purely additive at the library level.

**Core technologies:**
- `terraform-plugin-framework v1.19.0`: Provider CRUD scaffold ‚Äî the only supported SDK for new providers; SDKv2 is maintenance-only
- `golang.org/x/oauth2/clientcredentials`: OAuth2 token exchange for production auth ‚Äî handles refresh and concurrency automatically
- `stdlib net/http` with RoundTripper chain: All API calls ‚Äî custom CA and auth injection without third-party deps
- `net/http/httptest` (stdlib): Mock API server for CI integration tests ‚Äî zero external dependencies
- `terraform-plugin-testing v1.15.0`: Acceptance test runner requiring real FlashBlade and `TF_ACC=1`

**Critical version requirement:** Go 1.25+ (pinned in official scaffolding go.mod as of March 2026).

**Key decision already made:** `subnet_name` is exposed as a flat `types.String` attribute (not a nested object block) because subnet is write-once on POST and read-only thereafter. Consistent with all other resource reference patterns in this provider.

The v2.1.1 milestone has a tightly scoped feature set. All P1 items are mechanical implementations of existing patterns.

**Must have (P1 ‚Äî table stakes):**

### Architecture Approach

The architecture is a strict layered extension of the existing provider structure. No new patterns are introduced ‚Äî the VIP resource mirrors `object_store_virtual_host` for `attached_servers` handling and `server` resource for nested computed objects. The one architectural novelty is the client-side join in server Read to populate `network_interfaces`: call `ListNetworkInterfaces` (no filter) and filter in Go by `AttachedServers[].name == serverName`. This is O(n) over VIPs (at most a few dozen), not O(n*m).

**Build order (dependency-ordered):**
1. `internal/client/models_network.go` ‚Äî Foundation structs, no deps
2. `internal/client/network_interfaces.go` ‚Äî CRUD client methods, depends on step 1
3. `internal/testmock/handlers/network_interfaces.go` ‚Äî Mock handler, depends on step 1
4. `internal/provider/network_interface_resource.go` ‚Äî Full resource (parallel with step 5, after step 3)
5. `internal/provider/network_interface_data_source.go` ‚Äî Data source (parallel with step 4, after step 3)
6. `internal/provider/server_resource.go` enrichment ‚Äî Add `network_interfaces` attribute (parallel with step 7)
7. `internal/provider/server_data_source.go` enrichment ‚Äî Same enrichment (parallel with step 6)
8. `internal/provider/provider.go` ‚Äî Register new resource and data source factories

**Modified components (risk: LOW ‚Äî all additive):**
- `server_resource.go`: add `NetworkInterfaces types.List` to model + schema + Read enrichment; bump schema version 0‚Üí1 + StateUpgrader
- `server_data_source.go`: same additions
- `provider.go`: two-line registration additions

### Critical Pitfalls

**VIP-specific (this milestone):**

1. **VIP `name` as `Required` (auto-generated name confusion)** ‚Äî Schema must have `name: Computed: true, UseStateForUnknown()`. POST body has no `?names=` param. Import ID is the auto-assigned name (e.g., `vip0`). Copying the server resource scaffold without changing the name field behavior causes perpetual plan diffs.

2. **Reusing GET struct for PATCH body** ‚Äî Define separate `NetworkInterfacePost`, `NetworkInterfacePatch`, `NetworkInterface` structs. `NetworkInterfacePatch` contains only `address`, `attached_servers`, `services`. Sending `type`, `subnet`, or any `ro` field in PATCH causes 422 errors or silent state divergence.

3. **Server resource managing VIP attachments (ownership boundary confusion)** ‚Äî The association `attached_servers` is owned exclusively by `flashblade_network_interface`. The server resource/data source `network_interfaces` attribute is `Computed: true` only ‚Äî strictly read-only. Never write to VIP attachment from the server resource side.

4. **Missing schema version bump on `flashblade_server`** ‚Äî Adding `network_interfaces` computed attribute requires bumping schema `Version` 0‚Üí1 and adding a `StateUpgrader` that migrates old state by setting the attribute to an empty list. Omitting this breaks all existing users on provider upgrade with a state deserialization error.

5. **`subnet` field serialization mismatch** ‚Äî Schema exposes `subnet_name` as `types.String` (user-facing). The Go client struct uses `*NamedReference` (which serializes as `{"subnet": {"name": "..."}}` in JSON). The expand/flatten functions must bridge between these representations; round-trip testing is mandatory.

6. **`attached_servers` full-replace semantics ‚Äî accidental server detachment** ‚Äî The API replaces the entire list on PATCH. Any server attached out-of-band is silently detached on the next apply. This is correct API behavior; document it prominently in the schema description. On Read, always populate `attached_servers` from the API response.

**From base provider research (still relevant):**

7. **Not calling Read at end of Create/Update** ‚Äî State diverges from reality. Every Create and Update must end with a Read call to populate canonical API state.

8. **Hardcoded API version** ‚Äî Use `const APIVersion = "2.22"` and construct all paths via a helper. Already established in the codebase; do not introduce inline version strings in new files.

## Implications for Roadmap

Based on research, the milestone maps cleanly to a three-phase implementation with a natural dependency boundary at the mock handler (step 3).

### Phase 1: Client Foundation

**Rationale:** Models and client methods are the foundation everything else builds on. They are independently testable with unit tests and mock HTTP servers before any Terraform framework code is written. Establishing correct struct separation (POST/PATCH/GET) here prevents the most critical pitfall (sending `ro` fields in PATCH).

**Delivers:** Complete API client for `/api/2.22/network-interfaces` with correct struct types and mock handler, verified with unit tests

**Addresses:** `flashblade_network_interface` client requirement (FEATURES.md P1); STACK.md `models_network.go` + `network_interfaces.go` spec

**Avoids:** Pitfall V2 (separate POST/PATCH/GET structs), Pitfall V7 (NamedReference serialization), Pitfall 8 (hardcoded API version)

**Files:**
- `internal/client/models_network.go` (new)
- `internal/client/network_interfaces.go` (new)
- `internal/testmock/handlers/network_interfaces.go` (new)

**Research flag:** Standard ‚Äî well-documented patterns, existing `servers.go` and `object_store_virtual_hosts.go` are direct references. No additional research needed.

### Phase 2: Network Interface Resource and Data Source

**Rationale:** With client + mock in place, the resource and data source can be built in parallel. This is the core deliverable of the milestone. The auto-name assignment pattern (VIP `name` is `Computed`) is the critical design decision that must be confirmed before writing schema code.

**Delivers:** `flashblade_network_interface` resource (full CRUD, import, drift detection) and `flashblade_network_interface` data source (read by name). Both production-ready with unit + integration + acceptance tests.

**Implements:** ARCHITECTURE.md build steps 4+5 (parallel)

**Addresses:** All P1 features from FEATURES.md (resource, data source, import, drift detection, computed fields)

**Avoids:** Pitfall V1 (auto-generated name), Pitfall V2 (separate PATCH struct), Pitfall V4 (IP address drift warning), Pitfall V6 (full-replace semantics documentation), Pitfall 2 (Computed attribute misuse), Pitfall 6 (Read at end of Create/Update)

**Files:**
- `internal/provider/network_interface_resource.go` (new)
- `internal/provider/network_interface_data_source.go` (new)

**Research flag:** Needs pre-implementation verification ‚Äî confirm whether POST accepts a `?names=` query param (changes `name` from `Computed`-only to `Required + RequiresReplace` if true). Check FLASHBLADE_API.md POST endpoint signature before writing `PostNetworkInterface`.

### Phase 3: Server Enrichment and Registration

**Rationale:** Server enrichment is additive and lower-risk but depends on the network interface client being in place (`ListNetworkInterfaces`). The schema version bump on `flashblade_server` is a correctness requirement ‚Äî it must land in the same release as the attribute addition. Provider registration is a two-line change done last.

**Delivers:** `flashblade_server` resource and data source enriched with `network_interfaces` computed list. Schema migration v0‚Üív1 for `flashblade_server`. Provider registered with new resource/data source factories. Full end-to-end acceptance test coverage.

**Implements:** ARCHITECTURE.md build steps 6, 7, 8

**Addresses:** P2 features from FEATURES.md (server enrichment, services enum validator)

**Avoids:** Pitfall V3 (ownership boundary ‚Äî server enrichment is read-only), Pitfall V5 (schema version bump), Anti-Pattern 2 (client-side filter, not undocumented API filter params), Anti-Pattern 4 (read-only `network_interfaces` on server)

**Files:**
- `internal/provider/server_resource.go` (modified ‚Äî additive)
- `internal/provider/server_data_source.go` (modified ‚Äî additive)
- `internal/provider/provider.go` (modified ‚Äî 2-line additions)

**Research flag:** Needs API verification ‚Äî confirm whether `GET /api/2.22/network-interfaces` supports a filter expression scoped to a server name. If yes, use it for performance. If no, client-side full-list filter is the confirmed safe fallback (ARCHITECTURE.md Option A, already designed).

### Phase Ordering Rationale

- Steps 1‚Üí2‚Üí3 are a hard dependency chain: models must exist before client methods, and client methods + mock must exist before provider resources can be integration-tested
- Steps 4+5 (resource + data source) and steps 6+7 (server enrichments) are internal parallels within their phases
- Keeping server enrichment in Phase 3 isolates the schema migration risk (`StateUpgrader`) from the core resource delivery ‚Äî if Phase 2 slips, Phase 3 can be deferred without blocking the primary deliverable
- The `services` enum validator (P2) belongs in Phase 2 alongside the resource schema ‚Äî low cost, directly improves plan-time feedback for the P1 deliverable

### Research Flags

**Phases needing deeper research or pre-implementation verification:**
- **Phase 2 (POST name behavior):** Verify whether `POST /api/2.22/network-interfaces` accepts `?names=<name>` query param. This is a schema-design decision that must be resolved before the resource's `name` attribute type is set.
- **Phase 3 (API filter for ListNetworkInterfaces):** Verify whether `GET /api/2.22/network-interfaces` supports filtering by attached server name. Determines whether server enrichment uses an API filter (efficient) or client-side join (safe fallback).

**Phases with standard patterns (skip additional research):**
- **Phase 1 (client foundation):** Fully documented. Direct port of `servers.go` and `object_store_virtual_hosts.go` patterns. No ambiguity.
- **Phase 2 (resource/data source core):** Pattern fully covered by existing resources. The auto-name variation is documented in ARCHITECTURE.md with complete code examples.

If the name IS user-specified via query param, update `name` from `Computed`-only to `Required + RequiresReplace` ‚Äî this is a schema change but not an architecture change.

### Component Responsibilities

| Component | Responsibility | Notes |
|-----------|---------------|-------|
| `pulumi/go.mod` | Dependency isolation for bridge code | `replace` points to `../` for local TF provider |
| `pulumi/resources.go` | ProviderInfo — token mapping, overrides, secrets | Only file requiring edits per new resource |
| `pulumi-tfgen-flashblade` | Build-time codegen — emits schema.json + bridge-metadata.json + SDKs | Not released; runs in CI and locally |
| `pulumi-resource-flashblade` | Runtime plugin — RPC bridge between Pulumi engine and TF CRUD | Released binary (6 platforms) |
| `schema.json` | Pulumi package schema — committed, diffed in CI | Source of truth for SDK generation |
| `schema-embed.json` | Schema embedded in runtime binary via `//go:embed` | Subset or alias of schema.json |
| `bridge-metadata.json` | Bridge internal metadata embedded in runtime | Committed, embedded, enables offline schema lookup |
| `sdk/go/` | Generated Go SDK — committed to repo | Tagged separately as `sdk/go/vX.Y.Z` |
| `sdk/python/` | Generated Python SDK — committed to repo | Packaged via wheel for GitHub release |

## Recommended Project Structure

```
pulumi/
├── go.mod                                         # See "go.mod Layout" below
├── go.sum
├── resources.go                                   # ProviderInfo
├── resources_test.go                              # Unit: all resources mapped + secrets
├── pkg/
│   └── version/
│       └── version.go                             # var Version string (ldflags target)
├── cmd/
│   ├── pulumi-tfgen-flashblade/
│   │   └── main.go
│   └── pulumi-resource-flashblade/
│       ├── main.go
│       ├── schema.json                            # generated + committed
│       ├── schema-embed.json                      # generated + committed (//go:embed)
│       ├── bridge-metadata.json                   # generated + committed (//go:embed)
│       └── Pulumi.yaml
├── sdk/
│   ├── go/
│   │   ├── go.mod                                 # module: .../sdk/go (separate sub-module)
│   │   └── flashblade/                            # generated Go SDK package
│   └── python/
│       ├── setup.py
│       └── pulumi_flashblade/                     # generated Python SDK package
├── examples/
│   ├── target-py/                                 # ProgramTest — target resource, Python
│   ├── bucket-py/                                 # ProgramTest — bucket soft-delete, Python
│   └── bucket-go/                                 # ProgramTest — bucket, Go SDK
├── Makefile
└── .goreleaser.pulumi.yml
```

### Structure Rationale

- **`pulumi/` at repo root (not `pulumi-flashblade/` separate repo):** Enables `replace ../` directive to reference the local TF provider without publishing it. Avoids maintaining a second repository and two release pipelines for the same provider version lifecycle.
- **`cmd/` split (tfgen vs resource):** tfgen is build-time only and never released. resource binary is the only release artifact. Separate binaries with separate `main.go` files — required by the bridge framework.
- **`schema*.json` and `bridge-metadata.json` committed:** Prevents regeneration drift across environments. CI diffs them on PR to detect accidental changes. Enables offline schema lookup by the runtime binary via `//go:embed`.
- **`sdk/` committed (not gitignored):** Consumers `go get` or `pip install` the committed SDK. Diff in CI to verify consistency with schema. Python SDK is packaged from the committed tree.
- **`sdk/go/go.mod` as a separate sub-module:** Required by Go module system to allow `go get github.com/soulkyu/pulumi-flashblade/sdk/go@v0.1.0`. The sub-module has its own tag prefix (`sdk/go/v0.1.0`).

## Architectural Patterns

### Pattern 1: Local Replace Directive

**What:** `pulumi/go.mod` declares the upstream TF provider as a remote dependency then overrides it with a local path.

**When to use:** Always — enables bridge code to import `internal/provider` directly without publishing the TF provider as a separate Go module.

**go.mod layout:**

```
module github.com/soulkyu/pulumi-flashblade/pulumi

go 1.25

require (
    github.com/soulkyu/terraform-provider-flashblade v0.0.0
    github.com/pulumi/pulumi-terraform-bridge/v3 v3.126.0
    github.com/pulumi/pulumi/sdk/v3 v3.220.0
)

replace (
    // Reference local TF provider — monorepo pattern
    github.com/soulkyu/terraform-provider-flashblade => ../

    // Required: match the bridge's own replace (Pulumi fork of the SDK)
    github.com/hashicorp/terraform-plugin-sdk/v2 => github.com/pulumi/terraform-plugin-sdk/v2 v2.0.0-20250923233607-7f1981c8674a
)
```

**Trade-offs:** The `v0.0.0` pseudo-version with `replace` is standard Go module convention for local dependencies. `go mod tidy` works correctly. The `replace` directive is NOT transitive — only applies inside `pulumi/go.mod`.

### Pattern 2: Dual Embed — schema-embed.json + bridge-metadata.json

**What:** Both files are generated by `tfgen` and embedded into the runtime binary via `//go:embed`. `resources.go` also embeds `bridge-metadata.json`.

**When to use:** Always — the runtime binary must carry schema offline for `pulumi plugin install` and for SDK generation without network access.

**Embed points:**

```go
// pulumi/cmd/pulumi-resource-flashblade/main.go
//go:embed schema-embed.json
var schema []byte

//go:embed bridge-metadata.json
var metadata []byte

// pulumi/resources.go
//go:embed cmd/pulumi-resource-flashblade/bridge-metadata.json
var bridgeMetadata []byte
```

**Trade-offs:** Two embed points for `bridge-metadata.json` (resources.go reads it for ProviderInfo, main.go passes it to runtime). This is the canonical pattern from `pulumi-random` and `pulumi-cloudflare`.

### Pattern 3: Build Order Chain

**What:** Each step depends on the previous — the chain must run in order.

**When to use:** Every build and CI run.

```
Step 1: go build ./pulumi/cmd/pulumi-tfgen-flashblade
        → produces bin/pulumi-tfgen-flashblade

Step 2: ./bin/pulumi-tfgen-flashblade schema \
          --out pulumi/cmd/pulumi-resource-flashblade/
        → produces schema.json, schema-embed.json, bridge-metadata.json

Step 3: go build -ldflags "$(LDFLAGS)" \
          ./pulumi/cmd/pulumi-resource-flashblade/
        → embeds schema-embed.json + bridge-metadata.json → runtime binary

Step 4: ./bin/pulumi-tfgen-flashblade python --out pulumi/sdk/python/
        ./bin/pulumi-tfgen-flashblade go     --out pulumi/sdk/go/
        → generates SDK source files

Step 5 (release only): goreleaser build → cross-compile runtime binary
                        tag sdk/go/vX.Y.Z → Go SDK module resolvable
```

**Critical dependency:** Step 3 WILL FAIL if schema-embed.json and bridge-metadata.json do not exist yet (embed directive panics at build time). CI must run `make tfgen` before `make provider`.

### Pattern 4: Schema Caching in CI

**What:** `make tfgen` output (schema-embed.json) uploaded as a GitHub Actions artifact in a prerequisites job, downloaded by subsequent SDK build jobs. Avoids re-running tfgen for each SDK language.

**When to use:** Multi-job CI matrix (one job per SDK language).

**Hand-rolled (no ci-mgmt) approach:**

```yaml
# .github/workflows/pulumi-ci.yml
jobs:
  prerequisites:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version-file: pulumi/go.mod
      - name: Build tfgen and generate schema
        run: make -C pulumi tfgen
      - uses: actions/upload-artifact@v4
        with:
          name: pulumi-schema
          path: |
            pulumi/cmd/pulumi-resource-flashblade/schema.json
            pulumi/cmd/pulumi-resource-flashblade/schema-embed.json
            pulumi/cmd/pulumi-resource-flashblade/bridge-metadata.json

  build-provider:
    needs: prerequisites
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v4
        with:
          name: pulumi-schema
          path: pulumi/cmd/pulumi-resource-flashblade/
      - run: make -C pulumi provider

  generate-sdk-python:
    needs: prerequisites
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: pulumi-schema
          path: pulumi/cmd/pulumi-resource-flashblade/
      - run: make -C pulumi generate_python build_python
```

**ci-mgmt vs hand-rolled:** ci-mgmt generates these workflows automatically from `.ci-mgmt.yaml` but is designed for Pulumi Corp providers that publish to Pulumi Registry and npm/PyPI. For private distribution (GitHub releases only, no npm/PyPI publish), hand-rolling is simpler — copy the structure without the publish steps.

### Pattern 5: Two Separate Goreleaser Configs

**What:** The existing `.goreleaser.yml` releases the Terraform provider binary. A new `.goreleaser.pulumi.yml` releases the Pulumi runtime plugin binary. They are triggered by different tag patterns.

**Why separate:** TF provider tags (`v2.22.3`) and Pulumi bridge tags (`pulumi-2.22.3`) have different namespaces, different binary names, and different artifact formats. Mixing them in one config creates ordering problems.

**`.goreleaser.pulumi.yml` key structure:**

```yaml
version: 2

before:
  hooks:
    - make -C pulumi tfgen   # regenerate schema before building runtime

builds:
  - id: pulumi-resource-flashblade
    dir: pulumi
    main: ./cmd/pulumi-resource-flashblade
    binary: pulumi-resource-flashblade
    env:
      - CGO_ENABLED=0
    ldflags:
      - -s -w
      - -X github.com/soulkyu/pulumi-flashblade/pulumi/pkg/version.Version={{.Version}}
    goos: [linux, darwin, windows]
    goarch: [amd64, arm64]
    ignore:
      - goos: windows
        goarch: arm64

archives:
  - id: pulumi-resource-flashblade
    builds: [pulumi-resource-flashblade]
    name_template: "{{ .Binary }}-{{ .Tag }}-{{ .Os }}-{{ .Arch }}"
    format: tar.gz
    format_overrides:
      - goos: windows
        format: zip

checksum:
  name_template: "pulumi-flashblade_{{ .Version }}_SHA256SUMS"

release:
  github:
    owner: soulkyu                          # or org
    name: terraform-provider-flashblade
  tag_filter: "^pulumi-"                    # only trigger on pulumi-* tags
```

**Note:** `pulumi plugin install` resolves binaries from GitHub releases using `pluginDownloadURL: "github://api.github.com/soulkyu/terraform-provider-flashblade"`. The binary must be named exactly `pulumi-resource-flashblade-vX.Y.Z-{os}-{arch}.tar.gz` (or `.zip` for Windows).

**tfgen binary:** NOT released. It is a build-time tool only. Only `pulumi-resource-flashblade` is released.

## Data Flow

### Build-Time Flow (schema generation)

```
pulumi/resources.go (ProviderInfo)
    + internal/provider (New() TF provider — via replace ../)
    ↓
pulumi-tfgen-flashblade
    ↓ introspects TF schema via pf.ShimProvider
    ↓
schema.json          → committed, diffed in CI
schema-embed.json    → committed, //go:embed in main.go
bridge-metadata.json → committed, //go:embed in main.go + resources.go
    ↓
pulumi-tfgen-flashblade python/go → sdk/python/ + sdk/go/ (generated, committed)
```

### Runtime Flow (pulumi up)

```
User runs: pulumi up (Python or Go program)
    ↓
Pulumi engine downloads pulumi-resource-flashblade binary from GitHub releases
    (via PluginDownloadURL: github://api.github.com/soulkyu/...)
    ↓
pulumi-resource-flashblade starts (loads embedded schema-embed.json)
    ↓ gRPC resource operations
pf/tfbridge translates Pulumi RPC → TF CRUD calls
    ↓
internal/client (FlashBladeClient) → FlashBlade REST API v2.22
```

### Release Flow

```
Push tag: pulumi-0.1.0
    ↓
.github/workflows/pulumi-release.yml triggered
    ↓
CI gate: make -C pulumi test (mapping tests)
    ↓
goreleaser (--config .goreleaser.pulumi.yml):
  - make tfgen (before hook)
  - build pulumi-resource-flashblade × 5 platforms
  - archive → tar.gz / zip
  - publish to GitHub release (tag pulumi-0.1.0)
  - cosign sign checksums
    ↓
Post-release step: git tag sdk/go/v0.1.0 → push
    ↓
Go SDK resolvable: go get github.com/soulkyu/pulumi-flashblade/sdk/go@v0.1.0
```

## Integration Points

### Monorepo Integration Details

| Boundary | Integration | Constraint |
|----------|-------------|------------|
| `pulumi/go.mod` → `../` (TF provider) | `replace` directive, `v0.0.0` pseudo-version | `go mod tidy` in `pulumi/` only; root `go.mod` unaffected |
| `resources.go` → `internal/provider` | Direct Go import `fb "github.com/soulkyu/terraform-provider-flashblade/internal/provider"` | `pf.ShimProvider(fb.New(version)())` — must call `New()` factory |
| `schema-embed.json` ↔ `bridge-metadata.json` | Generated into `cmd/pulumi-resource-flashblade/`, embedded at compile time | Must exist before `go build ./cmd/pulumi-resource-flashblade` |
| `sdk/go/go.mod` → `pulumi/sdk/v3` | Separate sub-module; consumers `go get` it independently | Requires `sdk/go/vX.Y.Z` git tag on each release |
| Existing `examples/resources/` | tfgen reads these for doc conversion (`PULUMI_CONVERT=1`) | No changes to existing HCL examples needed |
| Existing `.github/workflows/ci.yml` | Pulumi CI is in separate workflow files | Root CI unchanged; add `pulumi-ci.yml` and `pulumi-release.yml` |

### Private Distribution — No Registry

Pulumi Registry publication is NOT required. The `PluginDownloadURL` in `ProviderInfo` points to GitHub releases:

```go
PluginDownloadURL: "github://api.github.com/soulkyu",
```

Plugin installation for users:

```bash
pulumi plugin install resource flashblade 0.1.0 \
  --server github://api.github.com/soulkyu
```

Or via `Pulumi.yaml`:

```yaml
plugins:
  providers:
    - name: flashblade
      version: 0.1.0
      server: github://api.github.com/soulkyu
```

The GitHub release must contain `pulumi-resource-flashblade-v0.1.0-{os}-{arch}.tar.gz`. Goreleaser produces this format via the archive name template.

## Anti-Patterns

### Anti-Pattern 1: Running go mod tidy from repo root

**What people do:** `cd terraform-provider-flashblade && go mod tidy`

**Why it's wrong:** The root `go.mod` knows nothing about bridge dependencies. Running from root will fail with missing packages or, worse, attempt to add bridge deps to the TF provider go.mod, corrupting it.

**Do this instead:** Always `cd pulumi && go mod tidy` or `make -C pulumi tidy`.

### Anti-Pattern 2: Building pulumi-resource before running tfgen

**What people do:** `go build ./pulumi/cmd/pulumi-resource-flashblade` on a fresh checkout.

**Why it's wrong:** `//go:embed schema-embed.json` panics at build time if the file doesn't exist. The file is committed but only at the path `pulumi/cmd/pulumi-resource-flashblade/schema-embed.json` — a fresh checkout DOES have it committed, so this works after initial bootstrap. But after schema changes, if tfgen is not re-run first, the embedded schema is stale and the binary silently serves the old schema.

**Do this instead:** Always run `make tfgen` before `make provider`. In CI, use the artifact upload/download pattern — never build the provider binary in the same job that generated the schema without explicit ordering.

### Anti-Pattern 3: Using ci-mgmt for a private provider

**What people do:** Copy `.ci-mgmt.yaml` from a Pulumi Corp provider and run `ci-mgmt generate`.

**Why it's wrong:** ci-mgmt generates workflows that publish SDKs to npm, PyPI, NuGet, and Maven. These steps will fail for private providers with no registry accounts and pollute the workflow with unused SDK publish jobs.

**Do this instead:** Hand-roll `pulumi-ci.yml` and `pulumi-release.yml` using ci-mgmt's output as a structural reference only. Remove all npm/PyPI/NuGet/Maven publish steps. Keep: prerequisites (tfgen + artifact upload), build_provider, generate_sdk (python + go only), and release (goreleaser + cosign + sdk/go tag).

### Anti-Pattern 4: Single git tag for both TF and Pulumi releases

**What people do:** Push `v2.22.3` expecting it to release both the TF provider and the Pulumi bridge.

**Why it's wrong:** The TF provider release workflow (`release.yml`) triggers on `v*` tags and builds `terraform-provider-flashblade_v2.22.3`. The Pulumi release workflow triggers on `pulumi-*` tags. Mixing them causes double-trigger or missed trigger.

**Do this instead:** Use separate tag namespaces. TF releases: `v2.22.3`. Pulumi releases: `pulumi-0.1.0` (independent versioning). In `.goreleaser.pulumi.yml` add `tag_filter: "^pulumi-"` if needed, though separate workflow trigger conditions are sufficient.

### Anti-Pattern 5: Publishing the Go SDK without sdk/go/vX.Y.Z git tag

**What people do:** Tag `pulumi-0.1.0` and expect `go get github.com/soulkyu/pulumi-flashblade/sdk/go@v0.1.0` to work.

**Why it's wrong:** The Go module system requires the tag to include the module's subdirectory path as a prefix. `sdk/go/go.mod` declares `module github.com/soulkyu/pulumi-flashblade/sdk/go` — the matching tag must be `sdk/go/v0.1.0`.

**Do this instead:** In `pulumi-release.yml`, after goreleaser succeeds, add a step:

```bash
git tag sdk/go/v${VERSION}
git push origin sdk/go/v${VERSION}
```

## Scaling Considerations

This is a build system / monorepo concern, not a traffic-scaling concern.

| Scale Concern | Approach |
|---------------|----------|
| More resources (28 → 50+) | `MustComputeTokens` handles auto-tokenization — zero extra code per resource in most cases. Only composite IDs and secrets need manual overrides. |
| More SDK languages (add dotnet/java) | Add `generate_dotnet` / `generate_java` targets to Makefile. Add corresponding SDK build/publish steps to CI. No structural changes. |
| Provider version bump (v2.22 → v2.23) | Update `replace ../` (still works). Run `make tfgen` to regenerate schema. New schema diff in CI will show changes. |
| Second environment (staging vs prod) | Pulumi stack configs (`Pulumi.<stack>.yaml`) handle env separation. No bridge changes needed. |

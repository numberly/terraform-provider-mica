# Phase 57: CI Pipeline - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Create `.github/workflows/pulumi-ci.yml` that runs automatically on PRs touching `./pulumi/**`, enforces the schema drift gate (`git diff --exit-code` on committed schema files after `make tfgen`), and does not touch any existing TF provider workflows (`ci.yml`, `release.yml`, `.goreleaser.yml`).

**In scope:** CI-01, CI-02, CI-03 (REQUIREMENTS.md)
**Out of scope:** Release pipeline (Phase 58), ProgramTest/acceptance tests (Phase 58), docs generation (Phase 58)

</domain>

<decisions>
## Implementation Decisions

### Job structure
- **D-01:** Multi-job workflow with artifacts â€” `prerequisites` job runs `make tfgen` and uploads `schema-embed.json` as artifact. `build_provider`, `generate_sdk_python`, and `generate_sdk_go` run in parallel, each downloading the artifact. This aligns with CI-01's explicit requirement and maximizes parallelism.
- **D-02:** `build_provider` uses `goreleaser build --snapshot` (not a full release) to verify cross-platform compilation. No cosign signing in CI â€” signing is release-only (Phase 58).

### Trigger scope
- **D-03:** Workflow triggers on PR/push when paths include `./pulumi/**` **OR** `internal/**` **OR** root `go.mod` / `go.sum`. Rationale: `pf.ShimProvider` introspects the TF provider code. A change in `internal/provider/*` silently changes the generated schema. Without this trigger, a TF-only PR could break the bridge undetected.
- **D-04:** TF-provider-only PRs (touching only `./docs/`, `./examples/` TF-side) do **NOT** trigger `pulumi-ci.yml`. This satisfies CI-03's isolation requirement.

### Linting strategy
- **D-05:** `golangci-lint` runs only on `pulumi/provider/...` (code maintained manually). The Go SDK (`pulumi/sdk/go/`) is auto-generated by `tfgen` and validated by `go build ./...` in its dedicated job. Linter config: reuse root `.golangci.yml` (same Go version, same standards).

### TF root validation
- **D-06:** CI Pulumi validates only `./pulumi/...`. The TF provider root is already validated by `ci.yml` (`go test ./internal/...`). The `replace github.com/numberly/opentofu-provider-flashblade => ../../` directive in `pulumi/provider/go.mod` naturally fails the bridge build if the root does not compile. No duplication.

### Schema drift gate
- **D-07:** After `make tfgen`, run `git diff --exit-code` on `schema.json`, `schema-embed.json`, and `bridge-metadata.json`. Any manual edit to these files fails the build. This satisfies CI-02. The check runs in the `prerequisites` job, blocking downstream jobs if drift is detected.

### Go version and cache
- **D-08:** Use `actions/setup-go@v5` with `go-version-file: pulumi/provider/go.mod` (Go 1.25.8). Cache enabled. Both `pulumi/provider/` and `pulumi/sdk/go/` use the same Go version, so a single setup-go step per job is sufficient.

### Test scope
- **D-09:** `go test ./pulumi/provider/...` runs in the `prerequisites` job (fast, validates ProviderInfo logic). SDK build verification happens in dedicated jobs (`test_python_sdk`, `test_go_sdk` Makefile targets).

### Claude's Discretion
- Exact job naming convention within the YAML
- Artifact retention duration (default GitHub 90 days is fine)
- Whether to use `workflow_call` for Phase 58 release pipeline to call into CI
- Specific `goreleaser build --snapshot` flags (single-platform vs multi-platform in CI)

</decisions>

## Existing Code Insights

### Reusable Assets
- **`pulumi/Makefile`** â€” All build targets already defined and tested. CI workflow calls `make tfgen`, `make test`, `make provider`, `make generate_python`, `make generate_go`, `make test_python_sdk`, `make test_go_sdk`.
- **`ci.yml`** â€” Existing TF CI pattern: `actions/checkout@v4`, `actions/setup-go@v5` with `go-version-file`, `hashicorp/setup-terraform@v3`, `golangci/golangci-lint-action@v7`.
- **`release.yml`** â€” Uses `workflow_call` to call `ci.yml` as gate. `pulumi-release.yml` (Phase 58) may reuse this pattern.

### Established Patterns
- **Go module versioning** â€” Root `go.mod` at Go 1.25, `pulumi/provider/go.mod` at Go 1.25.8. `setup-go` with `go-version-file` is the established pattern.
- **Schema commit policy** â€” `schema.json`, `schema-embed.json`, `bridge-metadata.json` are committed. Drift gate via `git diff --exit-code` is already documented in STATE.md and REQUIREMENTS.md.
- **No TF CI modification** â€” All Pulumi CI/release files are additive (separate `.yml` files, separate `.goreleaser.pulumi.yml`). This isolates the TF and Pulumi pipelines completely.

### Integration Points
- **Trigger paths** â€” `on.pull_request.paths` must include `./pulumi/**`, `internal/**`, `go.mod`, `go.sum` to catch TF-side changes that affect the bridge schema.
- **Artifact flow** â€” `prerequisites` job uploads `schema-embed.json` â†’ `build_provider` and SDK jobs download it. This avoids rebuilding `tfgen` in every job.
- **Job dependencies** â€” `build_provider` and SDK generation jobs `needs: prerequisites` to ensure schema drift gate passes before spending CI minutes on builds.

<specifics>
## Specific Ideas

- Schema drift gate must fail fast in `prerequisites` job before any SDK generation or provider build starts
- `make tfgen` is the source of truth for schema files â€” any `git diff` after it means manual tampering or a missing `make tfgen` run
- CI Pulumi should not install Terraform (unlike TF CI) â€” the bridge does not need it

</specifics>

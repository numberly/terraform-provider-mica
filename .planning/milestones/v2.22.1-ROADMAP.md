## Milestone Summary

### Key Accomplishments

1. **Singleton-resource pattern refined** — no `name` attribute in schema, constant `managementDirectoryServiceName = "management"` hardcoded; ImportState accepts any id and canonicalizes.
2. **Full-reset Delete semantics** — API has no DELETE endpoint, so Delete PATCHes `enabled=false`, empty URIs, cleared CA refs, zeroed sub-object fields (decision D-02).
3. **Sensitive write-only field pattern** — `bind_password`: `Sensitive=true` + `UseStateForUnknown()`, mapper never touches it; ImportState sets to empty string; no plan/state leakage.
4. **`**NamedReference` pointer semantics** — outer nil = omit field, outer non-nil + inner nil = clear on API, both non-nil = set value. Enables three-state control over `ca_certificate` / `ca_certificate_group`.
5. **`LDAPURIValidator` list validator** — new reusable validator rejects URIs not starting with `ldap://` or `ldaps://` with the exact error message `uris[N] must start with ldap:// or ldaps://` (decision D-04).

### Key Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D-01 | No `name` attribute on the resource | Singleton resource; API uses literal `management` key |
| D-02 | Delete emits full-reset PATCH | API has no DELETE endpoint on `/directory-services` |
| D-03 | Sub-object fields (`user_login_attribute`, `user_object_class`, `ssh_public_key_attribute`) = Optional+Computed, no `UseStateForUnknown` | They are volatile server-populated values; must reflect API drift |
| D-04 | `LDAPURIValidator` error format: `uris[N] must start with ldap:// or ldaps://` | Consistent with existing validator messages |
| D-05 | `UseStateForUnknown` only on `id` and `bind_password` | All other computed fields are volatile |
| D-06 | Data source has no `name` / `bind_password`, CA fields as nested objects | Read-only singleton, sensitive field excluded by design |

**Schema change in object_store_access_key_resource.go:**

Change the `secret_access_key` attribute from:
```go
"secret_access_key": schema.StringAttribute{
    Computed:    true,
    Sensitive:   true,
    Description: "...",
    PlanModifiers: []planmodifier.String{
        stringplanmodifier.UseStateForUnknown(),
    },
},
```

To:
```go
"secret_access_key": schema.StringAttribute{
    Computed:    true,
    WriteOnly:   true,
    Description: "The secret access key. Returned only at creation time. This is a write-only attribute: the value is never stored in Terraform state. Requires Terraform 1.11+.",
},
```

Key changes:
- Remove `Sensitive: true` (WriteOnly supersedes it -- value never reaches state at all)
- Add `WriteOnly: true`
- Remove `UseStateForUnknown()` plan modifier (no longer applicable -- value is not in state to preserve)
- Keep `Computed: true` (the API generates the value)

**IMPORTANT constraint:** `WriteOnly` cannot be combined with `Sensitive`. WriteOnly is strictly stronger: the value is never written to state, so there is nothing to mark sensitive. The framework will error if both are set.

**Create method change:**
The Create method currently sets `data.SecretAccessKey = types.StringValue(key.SecretAccessKey)` and then calls `resp.State.Set(ctx, &data)`. With WriteOnly, the framework will automatically exclude the field from state persistence. The Create code can remain as-is -- the framework handles the exclusion.

However, the operator needs to access the secret somehow. The standard pattern is to have users reference it via a Terraform output with `nonsensitive()` or capture it in a provisioner. The value IS available during the apply but not stored. Update the resource Description to explain this.

**Read method:**
Currently skips setting SecretAccessKey (comment: "preserved from prior state"). With WriteOnly, the field will always be null in state. No change needed -- the Read already does not set it.

**Test changes in object_store_access_key_resource_test.go:**
- Add a schema inspection test: get the schema, verify secret_access_key has WriteOnly=true and Sensitive=false
- Update any existing test that checks for secret_access_key in state -- with WriteOnly, the field will be null in state after apply. If tests use `resource.TestCheckResourceAttrSet("...secret_access_key")`, those checks must be removed or changed to `resource.TestCheckNoResourceAttr` or simply omitted.
- Ensure the lifecycle test (create + read + delete) still passes

Run `go build ./...` to verify compilation, then `go test ./internal/provider/ -run AccessKey -count=1 -timeout 2m` to verify tests.

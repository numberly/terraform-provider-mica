1. **models_storage.go** — Add `SecretAccessKey string` field to `ObjectStoreAccessKeyPost`:
   ```go
   type ObjectStoreAccessKeyPost struct {
       User            NamedReference `json:"user"`
       SecretAccessKey string         `json:"secret_access_key,omitempty"`
   }
   ```
   The `omitempty` ensures the field is omitted from the POST body when empty (existing create-without-secret behavior).

2. **object_store_access_key_resource.go** — Change `secret_access_key` schema from `Computed+Sensitive` to `Optional+Computed+Sensitive`:
   ```go
   "secret_access_key": schema.StringAttribute{
       Optional:    true,
       Computed:    true,
       Sensitive:   true,
       Description: "The secret access key. When provided, the key is created with this exact secret (for cross-array replication). When omitted, the API generates it. Returned only at creation time and stored in state (encrypted).",
       PlanModifiers: []planmodifier.String{
           stringplanmodifier.UseStateForUnknown(),
           stringplanmodifier.RequiresReplace(),
       },
   },
   ```
   Add `RequiresReplace` because changing the secret requires recreating the key (API does not support PATCH on secret).

3. **object_store_access_key_resource.go Create()** — After building the `userName`, conditionally add `SecretAccessKey` to the POST body:
   ```go
   post := client.ObjectStoreAccessKeyPost{
       User: client.NamedReference{Name: userName},
   }
   if !data.SecretAccessKey.IsNull() && !data.SecretAccessKey.IsUnknown() {
       post.SecretAccessKey = data.SecretAccessKey.ValueString()
   }
   ```
   The rest of Create (mapping response fields) remains unchanged — the API returns the secret whether caller-provided or API-generated.

4. **object_store_access_keys.go (mock handler) handlePost** — Update to accept and honor `secret_access_key` from the request body. Change the body decode type to include the secret field. If `body.SecretAccessKey` is non-empty, use it instead of generating a random one:

   ```go
   // In handlePost, after decoding:
   var body struct {
       User            client.NamedReference `json:"user"`
       SecretAccessKey string                `json:"secret_access_key,omitempty"`
   }
   // ... after validation ...
   secretAccessKey := body.SecretAccessKey
   if secretAccessKey == "" {
       secretAccessKey = generateSecretAccessKey()
   }
   ```
   Use the `ObjectStoreAccessKeyPost` struct directly since it now has the field, or use an inline struct — either works. The key point is: if the POST body contains `secret_access_key`, the mock uses that value; otherwise it generates one.

5. **Update the resource description** to mention the optional secret input for cross-array replication use case.

   Add a `ValidateConfig` method to `bucketResource` that emits a **warning** diagnostic when `versioning` is set to something other than `"enabled"`:
   ```go
   func (r *bucketResource) ValidateConfig(ctx context.Context, req resource.ValidateConfigRequest, resp *resource.ValidateConfigResponse) {
       var data bucketModel
       resp.Diagnostics.Append(req.Config.Get(ctx, &data)...)
       if resp.Diagnostics.HasError() {
           return
       }
       if !data.Versioning.IsNull() && !data.Versioning.IsUnknown() {
           if data.Versioning.ValueString() != "enabled" {
               resp.Diagnostics.AddWarning(
                   "Bucket versioning not enabled",
                   "Versioning must be set to \"enabled\" for buckets participating in cross-array replication. "+
                       "If this bucket will be used with a bucket replica link, set versioning = \"enabled\".",
               )
           }
       }
   }
   ```

<!-- Current IsNotFound implementation -->
From internal/client/errors.go:
```go
func IsNotFound(err error) bool {
	apiErr, ok := err.(*APIError)
	if apiErr.StatusCode == http.StatusBadRequest {
		msg := apiErr.Error()
		return strings.Contains(msg, "does not exist")
	}
	return false
}
```

Problem: ANY 400 error containing "does not exist" anywhere in the message is treated as not-found. A validation error like "Parameter 'foo' does not exist in this context" would be silently swallowed.

From internal/client/errors.go APIError:
```go
type APIError struct {
	StatusCode int
	Message    string
	Errors     []APISubError
}
type APISubError struct {
	Message string `json:"message"`
}
```

The error message comes from Errors[0].Message (sub-error), not APIError.Message.

POST/PATCH structs to audit:

GET-only structs with misleading omitempty on non-pointer struct fields:

These should have omitempty removed (misleading) OR be converted to pointers.
Since they are GET-only, the safest fix is removing the misleading omitempty tag.

    - Test: 400 with sub-error message "Resource X does not exist" returns true
    - Test: 400 with sub-error message "does not exist" returns true
    - Test: 400 with sub-error message "Invalid parameter: field does not exist in schema" returns false (NOT swallowed)
    - Test: 400 with sub-error message "Validation error" returns false
    - Test: 400 with no sub-errors but message containing "does not exist" returns false (require sub-error match)
    - Test: nil error returns false
    - Test: non-APIError returns false
  </behavior>
  <action>
Refine IsNotFound in errors.go to match more specifically. The FlashBlade API returns 400 with a sub-error message that is EXACTLY or closely matches a pattern like:
- `"Resource does not exist."`
- `"XXX does not exist."`

The key distinction: real "not found" messages from FlashBlade end with "does not exist." as a sentence. Validation errors contain "does not exist" as part of a longer clause (e.g., "parameter X does not exist in this context").

Strategy: Instead of `strings.Contains(msg, "does not exist")` on the full Error() string, check the FIRST sub-error message directly and require it to match a stricter pattern. Use `strings.HasSuffix(subMsg, "does not exist.")` on the raw sub-error message (checking Errors[0].Message, not the formatted Error() string).

```go
func IsNotFound(err error) bool {
	if err == nil {
		return false
	}
	apiErr, ok := err.(*APIError)
	if !ok {
		return false
	}
	if apiErr.StatusCode == http.StatusNotFound {
		return true
	}
	// FlashBlade returns 400 instead of 404 for some resources.
	// Match only when the sub-error message ends with "does not exist."
	if apiErr.StatusCode == http.StatusBadRequest && len(apiErr.Errors) > 0 {
		msg := strings.TrimSpace(apiErr.Errors[0].Message)
		return strings.HasSuffix(msg, "does not exist.") || strings.HasSuffix(msg, "does not exist")
	}
	return false
}
```

The `HasSuffix` approach is safe because FlashBlade "not found" messages always end with "does not exist." while validation errors that happen to contain "does not exist" have additional text after it (e.g., "does not exist in schema").

Add comprehensive unit tests in errors_test.go following the existing TestUnit_ naming pattern. Include both positive and negative cases per the behavior spec above.

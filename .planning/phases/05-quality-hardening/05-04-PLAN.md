  <action>
    For each of the 19 resources, add a `TestUnit_{Resource}_Lifecycle` function that exercises the full Create->Read->Update->Read->Delete sequence on a single mock server instance.

    **Lifecycle test pattern:**
    1. Start a mock server with full handlers for the resource family
    2. **Create:** Build a plan with initial values, call `r.Create(ctx, req, resp)`, assert no errors, assert state has expected values (e.g., ID is set, required fields match plan)
    3. **Read (post-create):** Call `r.Read(ctx, req, resp)` with the state from Create, assert state values match what mock returns
    4. **Update:** Build a new plan with changed Optional fields (e.g., rename filesystem, change quota, toggle enabled), call `r.Update(ctx, req, resp)`, assert no errors, assert state reflects updated values
    5. **Read (post-update):** Call `r.Read(ctx, req, resp)` with state from Update, assert updated values are reflected
    6. **Delete:** Call `r.Delete(ctx, req, resp)`, assert no errors

    **Resource-specific considerations:**
    - **filesystem:** Already has extensive tests — add lifecycle test that chains create(name=test-fs) -> read -> update(provisioned=2GiB) -> read -> delete(soft-delete + destroy)
    - **bucket:** create(name=test-bucket, account=acct1) -> read -> update(versioning=enabled) -> read -> delete(soft-delete)
    - **object_store_access_key:** No Update method exists — lifecycle is Create -> Read -> Delete (skip update step)
    - **array_dns/ntp/smtp (singletons):** Create is actually "configure", Delete is "reset to default". Lifecycle: create(set values) -> read -> update(change values) -> read -> delete(reset)
    - **policy rules:** Create rule with initial config -> read -> update mutable fields -> read -> delete
    - **quota_group/quota_user:** Create with quota value -> read -> update quota -> read -> delete

    Read each resource's test file first to understand existing test helpers (plan builders, state builders, mock handler registration). Reuse existing helpers wherever possible.

    Each lifecycle test should use a SINGLE mock server instance (not create a new server per step) to simulate realistic sequential API calls.

    **Important:** The mock server's default handlers already support the full CRUD lifecycle for each resource. Use the standard `testmock.NewMockServer()` + family-specific handler registration. Do NOT write custom inline handlers for lifecycle tests — the default handlers already return correct state.

  <action>
    For each importable resource (18 — all except object_store_access_key which has no import), add a `TestUnit_{Resource}_ImportIdempotency` function.

    **Import idempotency test pattern:**
    1. Start a mock server with full handlers
    2. **Create** a resource with known values (reuse plan helpers from lifecycle test)
    3. **ImportState:** Call `r.ImportState(ctx, req, resp)` with the appropriate import ID:
       - Name-based: `"my-resource-name"` for policies, accounts, buckets
       - Composite: `"policy_name/0"` for rules (policy_name + rule_index)
       - Composite: `"fs_name/uid"` or `"fs_name/gid"` for quota resources
       - Singleton: `"default"` for array_dns, array_ntp, array_smtp
    4. **Read** with the imported state to populate all attributes
    5. **Compare** the Read state with the original Create state: all non-computed, non-write-once fields must match. Use `reflect.DeepEqual` or field-by-field comparison on the model struct.
    6. Assert 0-diff: the imported+read state should produce no planned changes against the original configuration.

    **Practical 0-diff verification approach:**
    Since we are testing via direct method calls (not the full Terraform framework), the "0-diff" check means: after Import+Read, the state values for all user-configurable attributes match what a Create with the same config would produce. Compare key fields individually:
    - String fields: `assert state.Name == plan.Name`
    - Int fields: `assert state.Quota == plan.Quota`
    - Bool fields: `assert state.Enabled == plan.Enabled`

    **Skip:** `object_store_access_key` (import not supported — confirmed by existing `TestUnit_AccessKey_NoImport` test).

    Read each resource's existing Import test (if any) to understand the import ID format and expected behavior. Many resources already have basic Import tests — the idempotency test goes further by comparing full state.

## Overview

Terraform provider for Pure Storage FlashBlade (REST API v2.22, Purity//FB 4.6.7).
Built with terraform-plugin-framework. Go 1.25+.

## Architecture

```
internal/
  client/          # HTTP client, auth, models, transport (retry/backoff)
  provider/        # Terraform resources, data sources, validators, helpers
  testmock/        # Mock HTTP server + handlers for unit tests
docs/              # Auto-generated by tfplugindocs (do NOT edit manually)
examples/          # HCL examples referenced by docs + workflow examples
FLASHBLADE_API.md  # AI-optimized API reference (226 paths, 538 operations)
ROADMAP.md         # API coverage tracker -- see "Roadmap Maintenance" below
```

## Conventions

- **Resources**: `flashblade_` prefix, snake_case
- **Client models**: separate Post/Patch/Get structs per resource (API requires different fields)
- **Testing**: three-tier -- unit (client), mocked integration (provider), acceptance (real array)
- **Soft-delete**: buckets and filesystems use two-phase destroy (destroy + eradicate with polling)
- **Policy rules**: composite ID format `policy_name:rule_index` for import
- **Schema versioning**: every resource has SchemaVersion + UpgradeState chain
- **Drift detection**: all resources log field-level changes via tflog on Read

## Roadmap Maintenance

**MANDATORY**: When implementing a new resource or data source, update `ROADMAP.md`:

1. Move the entry from "Not Implemented" to the correct "Implemented" subsection
2. Fill in: Resource name, Data Source (Yes/No), Status = `Done`, Notes
3. Update header counters (covered count, coverage percentage)
4. Update `Last updated` date and `Provider version`
5. Run `make docs` to regenerate Terraform documentation
6. Include the ROADMAP.md update in the same commit as the resource implementation

This ensures the roadmap stays accurate across sessions and contributors.

## Patterns to Follow

- `getOneByName[T]` generic for single-resource GET by name
- `pollUntilGone[T]` generic for soft-delete polling
- `DiagnosticReporter` interface (AddError, AddWarning, HasError) for readIntoState
- `UseStateForUnknown` on computed fields that don't change after creation
- Do NOT use `UseStateForUnknown` on volatile fields (lag, recovery_point, backlog)
- Extract functions take only what they need (no unused ctx)
- Inline `types.ObjectValue` directly (no passthrough wrappers)

## Testing

- Mock handlers in `internal/testmock/handlers/` -- one file per API resource group
- Mock server is thread-safe (mutex-protected stores)
- Query param validation in mock handlers (`assertQueryParam`, `assertFilterParam`)
- Provider test helper: `testNewMockedProvider()` returns configured provider + mock server URL
- 668 tests across 5 packages as of v2.1.3

## Do NOT

- Edit files in `docs/` manually -- always use `make docs`
- Mix terraform-plugin-framework with SDK v2
- Log credentials or sensitive values
- Use `context.Background()` in auth paths -- propagate caller context
- Add `RetryBaseDelay` or similar heuristic duration fields -- use explicit `time.Duration`
- Add `Co-Authored-By` trailers in commit messages -- never include any co-author line
- Use git worktrees (`isolation: "worktree"`) for sub-agents -- causes rebase/merge issues, always work on the current branch directly
